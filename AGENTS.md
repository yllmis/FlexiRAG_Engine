# AGENTS 执行约定（FlexiRAG Engine）

本文档用于约束本仓库中 AI 编码助手的默认执行行为。

## 1. 语言与沟通

- 所有回复、注释、说明统一使用中文。
- 每次代码改动后必须明确说明：
  - 为什么要这么写这些方法，运用到了哪些知识点
  - 为什么修改
  - 验证结果
  - 提出适当的相关问题让用户回答
  - 修改代码时，加入必要的注释，比如一些方法和结构体一些元素等
- 用户提问前端相关时需要明确：用户没有任何前端基础。请不要直接给用户全部代码。请先告诉用户，这个页面在 Vue 3 里应该拆分成哪几个组件（Component）？它们之间的数据是怎么传递的？画个文本结构图给我用户。

## 2. 改动触发规则

- 默认只提供方案与建议，不直接改代码。
- 仅当用户明确表达类似“修改代码 / 写代码 / 直接改”时，才执行文件改动。

## 3. 环境与命令

- 按 macOS 思路优先提供路径与命令。
- 遇到服务启动失败，优先检查容器、端口、环境变量与数据库扩展状态。

## 4. 测试与验收（强约束）

- 不能“强行通过”，必须通过真实可执行检验验证逻辑与架构。
- 每次除了文档改动后至少执行（比如修改了某些功能）：
  - `go test ./...`
- 若当前改动缺少对应测试文件，必须补充单测。
- 测试文件命名规则：按被测文件命名（例如 `handler.go` 对应 `handler_test.go`），不要按单个方法命名测试文件。

## 5. 提交与推送

- 提交信息使用中文规范前缀：`feat:`、`fix:`、`release:`、`chore:` 等。
- 推送前提交中总结变更，再执行推送。
- 推送前更新[READEME.md](README.md)

## 6. API/架构改动约定

- 变更遵循分层：Router -> Handler -> Repository/Port。
- 先补 Port 接口，再补基础设施实现，再暴露 Handler 与 Router。
- RESTful 风格优先：
  - 列表查询使用 `GET /resources`
  - 局部更新使用 `PUT /resources/:id`

## 7. 文档维护

- 当用户新增执行约束时，应同步更新本文档。
- 每次实际代码改动后，需将本次变更摘要写入独立进度文档或对应设计文档，不写入 README，便于新开窗口后快速了解项目状态与开发进度。
- 文档遵循“单一真实状态”：仅保留当前有效结论；已废弃/已回滚且会造成歧义的描述应删除。
- 可保留历史阶段信息，但需确保不会与当前状态冲突。
- 若规则冲突，以用户最新明确要求为准。

## 8. 模块交付与考核

- 每写完一个模块后，必须先进行人工 review，再进入下一模块开发。
- 每个模块完成后，需提炼该模块的重要知识点并形成考核题，对用户进行针对性考核与回顾。
- 模块完成后，需要提问是否需要提交到本地或者推送到远程，在提出需要后请遵循规范提交/推送
